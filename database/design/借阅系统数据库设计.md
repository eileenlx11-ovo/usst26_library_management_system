# ③ 数据库设计负责人 B：借阅系统数据库设计

## 一、负责内容

本部分主要负责图书馆管理系统中的 **借阅系统数据库设计**，包括读者信息管理、借阅记录管理、借阅规则管理以及逾期罚款管理。

主要涉及以下数据表：

- `Reader`：读者表
- `BorrowRecord`：借阅记录表
- `Fine`：罚款表
- `Rule`：借阅规则表

核心业务逻辑包括：

- 记录借阅日期
- 自动计算应还日期
- 判断是否续借
- 计算超期天数
- 根据借阅规则自动生成罚款

## 二、借阅系统 E-R 图设计

### 1. 实体说明

#### 1.1 Reader 读者实体

用于保存读者的基本信息。

主要属性：

- 读者编号
- 姓名
- 性别
- 电话
- 读者类型
- 注册日期
- 状态

#### 1.2 BorrowRecord 借阅记录实体

用于记录读者每一次借书和还书的情况。

主要属性：

- 借阅记录编号
- 读者编号
- 图书编号
- 规则编号
- 借阅日期
- 应还日期
- 实际归还日期
- 是否续借
- 超期天数
- 借阅状态

#### 1.3 Fine 罚款实体

用于记录因逾期归还图书产生的罚款信息。

主要属性：

- 罚款编号
- 借阅记录编号
- 罚款金额
- 是否缴纳
- 生成日期
- 缴纳日期

#### 1.4 Rule 借阅规则实体

用于保存不同读者类型对应的借阅规则。

主要属性：

- 规则编号
- 读者类型
- 最大借阅天数
- 最大借阅数量
- 每日逾期费用
- 最大续借次数
- 续借天数

### 2. 实体联系说明

1. 一个读者可以产生多条借阅记录，一条借阅记录只属于一个读者。  
   因此，`Reader` 与 `BorrowRecord` 是 **一对多关系**。

2. 一条借阅记录可能产生一条罚款记录，也可能不产生罚款记录。  
   因此，`BorrowRecord` 与 `Fine` 是 **一对零或一关系**。

3. 一条借阅规则可以适用于多条借阅记录，一条借阅记录对应一条借阅规则。  
   因此，`Rule` 与 `BorrowRecord` 是 **一对多关系**。

### 3. 借阅 E-R 图文字表示

  1 ─────── 0..1

## 四、SQL 建表语句

以下 SQL 以 MySQL 为例。

### 1. 读者表 Reader

```sql
create table Reader (
reader_id int primary key auto_increment,
reader_name varchar(50) not null,
gender varchar (10),
phone varchar(20),
reader_type  varchar(20) not null,
register_date date not null,
status varchar (20) default '正常'
);
```

### 2. 借阅规则表 Rule

```sql
CREATE TABLE `Rule` (
    rule_id INT PRIMARY KEY AUTO_INCREMENT,
    reader_type VARCHAR(20) NOT NULL,
    max_borrow_days INT NOT NULL,
    max_borrow_count INT NOT NULL,
    fine_per_day DECIMAL(10,2) NOT NULL,
    max_renew_times INT NOT NULL,
    renew_days INT NOT NULL
);
```

### 3. 借阅记录表 BorrowRecord

```sql
CREATE TABLE BorrowRecord (
    borrow_id INT PRIMARY KEY AUTO_INCREMENT,
    reader_id INT NOT NULL,
    book_id INT NOT NULL,
    rule_id INT NOT NULL,
    borrow_date DATE NOT NULL,
    due_date DATE NOT NULL,
    return_date DATE,
    is_renewed BOOLEAN DEFAULT FALSE,
    overdue_days INT DEFAULT 0,
    borrow_status VARCHAR(20) DEFAULT '借阅中',

    CONSTRAINT fk_borrow_reader
        FOREIGN KEY (reader_id)
        REFERENCES Reader(reader_id),

    CONSTRAINT fk_borrow_rule
        FOREIGN KEY (rule_id)
        REFERENCES `Rule`(rule_id)
);
```

### 4. 罚款表 Fine

```sql
CREATE TABLE Fine (
    fine_id INT PRIMARY KEY AUTO_INCREMENT,
    borrow_id INT NOT NULL UNIQUE,
    fine_amount DECIMAL(10,2) NOT NULL,
    is_paid BOOLEAN DEFAULT FALSE,
    create_date DATE NOT NULL,
    pay_date DATE,

    CONSTRAINT fk_fine_borrow
        FOREIGN KEY (borrow_id)
        REFERENCES BorrowRecord(borrow_id)
);
```

## 五、借阅记录 SQL 设计

### 1. 插入借阅规则

```sql
INSERT INTO `Rule`
(reader_type, max_borrow_days, max_borrow_count, fine_per_day, max_renew_times, renew_days)
VALUES
('学生', 30, 5, 0.50, 1, 15),
('教师', 60, 10, 0.30, 2, 30);
```

### 2. 插入读者信息

```sql
INSERT INTO Reader
(reader_name, gender, phone, reader_type, register_date, status)
VALUES
('张三', '男', '13800000001', '学生', '2026-05-01', '正常'),
('李四', '女', '13800000002', '教师', '2026-05-02', '正常');
```

### 3. 新增借阅记录

假设学生借阅规则的 `rule_id = 1`。

```sql
INSERT INTO BorrowRecord
(reader_id, book_id, rule_id, borrow_date, due_date, is_renewed, borrow_status)
VALUES
(1, 1001, 1, '2026-05-01', DATE_ADD('2026-05-01', INTERVAL 30 DAY), FALSE, '借阅中');
```

说明：

```text
应还日期 = 借阅日期 + 最大借阅天数
```

### 4. 续借操作 SQL

```sql
UPDATE BorrowRecord
SET 
    due_date = DATE_ADD(due_date, INTERVAL 15 DAY),
    is_renewed = TRUE
WHERE borrow_id = 1
  AND is_renewed = FALSE
  AND borrow_status = '借阅中';
```

说明：

- 若未续借，则允许续借一次。
- 续借后应还日期增加 15 天。
- `is_renewed` 改为 `TRUE`。

### 5. 归还图书 SQL

```sql
UPDATE BorrowRecord
SET
    return_date = '2026-06-20',
    overdue_days = GREATEST(DATEDIFF('2026-06-20', due_date), 0),
    borrow_status = '已归还'
WHERE borrow_id = 1;
```

说明：

```text
超期天数 = 实际归还日期 - 应还日期
如果没有超期，则超期天数为 0。
```

## 六、自动罚款设计

当读者归还图书时，如果实际归还日期超过应还日期，系统应自动计算逾期天数，并根据每日罚款金额生成罚款记录。

罚款计算公式：

```text
罚款金额 = 超期天数 × 每日逾期费用
```

### 自动生成罚款触发器

```sql
DELIMITER $$

CREATE TRIGGER trg_auto_create_fine
AFTER UPDATE ON BorrowRecord
FOR EACH ROW
BEGIN
    DECLARE v_fine_per_day DECIMAL(10,2);
    DECLARE v_fine_amount DECIMAL(10,2);

    IF NEW.return_date IS NOT NULL
       AND NEW.overdue_days > 0
       AND OLD.return_date IS NULL THEN

        SELECT fine_per_day
        INTO v_fine_per_day
        FROM `Rule`
        WHERE rule_id = NEW.rule_id;

        SET v_fine_amount = NEW.overdue_days * v_fine_per_day;

        INSERT INTO Fine
        (borrow_id, fine_amount, is_paid, create_date)
        VALUES
        (NEW.borrow_id, v_fine_amount, FALSE, CURDATE());

    END IF;
END$$

DELIMITER ;
```

### 示例

如果某读者：

```text
借阅日期：2026-05-01
应还日期：2026-05-31
实际归还日期：2026-06-05
每日罚款：0.50 元
```

则：

```text
超期天数 = 5 天
罚款金额 = 5 × 0.50 = 2.50 元
```

系统会自动在 `Fine` 表中生成一条罚款记录。

## 七、范式分析

### 1. Reader 表范式分析

`Reader(reader_id, reader_name, gender, phone, reader_type, register_date, status)`

主键为：

```text
reader_id
```

函数依赖：

```text
reader_id → reader_name, gender, phone, reader_type, register_date, status
```

分析：

- 各字段都是不可再分的原子值，满足第一范式。
- 主键为单属性，不存在部分函数依赖，满足第二范式。
- 非主属性不依赖其他非主属性，满足第三范式。

因此，`Reader` 表满足 **第三范式 3NF**。

### 2. Rule 表范式分析

`Rule(rule_id, reader_type, max_borrow_days, max_borrow_count, fine_per_day, max_renew_times, renew_days)`

主键为：

```text
rule_id
```

函数依赖：

```text
rule_id → reader_type, max_borrow_days, max_borrow_count, fine_per_day, max_renew_times, renew_days
```

分析：

- 各属性均为原子值，满足第一范式。
- 主键为单属性，不存在部分函数依赖，满足第二范式。
- 借阅天数、最大借阅数量、逾期费用等规则信息均直接依赖于规则编号，不存在传递依赖，满足第三范式。

因此，`Rule` 表满足 **第三范式 3NF**。

### 3. BorrowRecord 表范式分析

`BorrowRecord(borrow_id, reader_id, book_id, rule_id, borrow_date, due_date, return_date, is_renewed, overdue_days, borrow_status)`

主键为：

```text
borrow_id
```

函数依赖：

```text
borrow_id → reader_id, book_id, rule_id, borrow_date, due_date, return_date, is_renewed, overdue_days, borrow_status
```

分析：

- 每个字段都不可再分，满足第一范式。
- 主键为单属性，不存在部分函数依赖，满足第二范式。
- 表中记录的是一次具体借阅行为，读者编号、图书编号、借阅日期、应还日期、归还日期等信息均直接依赖于借阅记录编号，不存在明显传递依赖，满足第三范式。

说明：

- `overdue_days` 可以由 `return_date` 和 `due_date` 计算得到。
- 为了方便查询和统计，系统中保留该字段。
- 该字段属于可计算冗余字段，但不影响本系统的基本业务实现。

因此，`BorrowRecord` 表基本满足 **第三范式 3NF**。

### 4. Fine 表范式分析

`Fine(fine_id, borrow_id, fine_amount, is_paid, create_date, pay_date)`

主键为：

```text
fine_id
```

函数依赖：

```text
fine_id → borrow_id, fine_amount, is_paid, create_date, pay_date
```

分析：

- 各字段都是原子值，满足第一范式。
- 主键为单属性，不存在部分函数依赖，满足第二范式。
- 罚款金额、是否缴纳、生成日期、缴纳日期均直接依赖于罚款编号，不存在传递依赖，满足第三范式。

因此，`Fine` 表满足 **第三范式 3NF**。

## 八、小结

本部分完成了图书馆管理系统中借阅模块的数据库设计。通过 `Reader`、`BorrowRecord`、`Fine` 和 `Rule` 四张表，实现了读者借阅、归还、续借、逾期判断和自动罚款等核心功能。数据库结构清晰，表之间通过主键和外键建立联系，并基本满足第三范式要求，能够支持图书馆借阅业务的正常运行。

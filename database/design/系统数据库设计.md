## 二、范式分析

所有关系模式均满足**第三范式（3NF）**。

### 2.1 第一范式（1NF）

所有属性均为不可分割的原子值：
- 姓名、电话、地址等均为单值属性
- 不存在集合属性或重复组

### 2.2 第二范式（2NF）

所有非主属性完全函数依赖于主键：
- 各表主键均为单属性（自增 ID），不存在复合主键
- 因此不可能出现部分函数依赖，自然满足 2NF

### 2.3 第三范式（3NF）

不存在非主属性对主键的传递依赖：

| 表 | 分析 |
|------|------|
| Book | author_id、category_id、publisher_id 是外键引用而非冗余数据，作者/分类/出版社信息独立存储在各自表中，不存在传递依赖 |
| BorrowRecord | 通过 rule_id 关联规则表，未直接存储 max_borrow_days、fine_per_day 等规则内容；通过 reader_id 关联读者表，未冗余存储读者姓名 |
| Fine | 通过 borrow_id 关联借阅记录，罚款金额由触发器根据规则计算生成，未冗余存储逾期天数或费率 |
| Reader | reader_type 用于关联 Rule 表的借阅规则，读者自身属性（姓名、电话、状态）直接依赖于 reader_id |
| User | 通过 reader_id 外键关联读者表，未冗余存储读者姓名或类型；角色、密码等属性直接依赖于 user_id |
| Author/Publisher/Category | 结构简单，属性直接依赖于主键，无传递依赖 |

### 2.4 设计取舍说明

以下字段属于**有意的冗余设计**，用于提升查询性能和业务准确性：

- `BorrowRecord.overdue_days`：归还时一次性计算并固化，避免每次查询都重新计算（且 due_date 可能因续借变化）
- `Book.available_count`：由触发器自动维护，避免每次查询都 COUNT 借阅记录
- `BorrowRecord.due_date`：借阅时根据规则计算写入，续借时直接修改此字段，保证历史记录准确

这些冗余通过触发器和存储过程保证数据一致性，属于合理的反范式优化。

## 四、大关系存储策略分析

### 4.1 数据规模分析

在本系统的 10 张表中，数据增长速率差异明显：

| 表 | 增长特征 | 预估规模（运行3年） |
|------|----------|---------------------|
| BorrowRecord | 每次借书产生一条记录，增长最快 | 10万+ 行 |
| Fine | 随逾期借阅产生，约占借阅量的 15%~20% | 1.5万+ 行 |
| Reader | 每学期新增，增长平稳 | 数千行 |
| Book | 馆藏增长较慢 | 数千行 |
| Author/Publisher/Category | 基础数据，几乎不增长 | 数百行 |

**结论：BorrowRecord 是系统中的大关系**，Fine 表次之。

### 4.2 针对大关系的存储优化策略

#### （1）索引加速

针对 BorrowRecord 的高频查询场景建立了 4 个索引：

- `idx_borrow_reader`：按读者查历史借阅记录
- `idx_borrow_book`：按图书查借阅情况
- `idx_borrow_status`：按状态筛选（借阅中/逾期/已归还）
- `idx_borrow_date`：按时间范围进行统计报表查询

Fine 表建立了 `idx_fine_paid` 索引用于快速查询未缴罚款。

#### （2）冗余字段设计（反范式优化）

为避免对大关系进行高代价的实时计算，引入以下冗余字段：

- `BorrowRecord.overdue_days`：归还时一次性计算并固化逾期天数，避免每次查询都执行 `DATEDIFF(return_date, due_date)` 计算
- `BorrowRecord.due_date`：借阅时根据规则写入，续借时直接修改，避免每次都 JOIN Rule 表重新计算
- `Book.available_count`：由触发器自动维护，避免每次展示可借数量时都 `COUNT` BorrowRecord 中未归还的记录

这些冗余字段通过触发器保证与源数据的一致性。

#### （3）历史数据归档策略（适用于生产环境）

当 BorrowRecord 数据量超过 10 万行时，可采用以下策略：

- **按时间分区**：按 `borrow_date` 年份做 RANGE 分区，将历史年份的已归还记录分到独立分区，缩小活跃查询的扫描范围
- **冷热分离**：将 `borrow_status = '已归还'` 且超过 2 年的记录迁移到归档表 `BorrowRecord_Archive`，主表只保留活跃数据

当前测试环境数据量较小（150 条），暂未实际启用分区，但索引和冗余字段的设计已为大数据量做好准备。
